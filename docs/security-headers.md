# Security headers — portable config (enforced *later*, not on Pages)

This repo ships **two** layers of HTTP-security configuration. Be precise about
which one is live, because they are enforced in different places:

| Layer | File | Enforced where | Status today |
|---|---|---|---|
| **Enforced now** | `<meta http-equiv="Content-Security-Policy">` in every page (generated by `scripts/build-csp.mjs`) | The browser, on GitHub Pages | **Live** |
| **Enforced later** | [`/_headers`](../_headers) (+ the Cloudflare snippet below) | A header-capable host/proxy in front of the site | **Inert** until such a host exists |

## Why two layers — the GitHub Pages limitation (stated plainly)

GitHub Pages serves a **fixed** set of response headers and lets you set **none**
of your own. So on Pages today:

- ✅ **CSP works** — but only the subset deliverable via `<meta http-equiv>`.
- ❌ **`frame-ancestors`** is **ignored inside a `<meta>` tag** (it is a
  header-only directive). Clickjacking protection therefore is **not** active on
  Pages. `X-Frame-Options: DENY` (also a header) is likewise unavailable.
- ❌ **`report-uri` / `report-to`** and document **sandbox** directives are also
  header-only and have no effect in `<meta>`.
- ❌ **HSTS, `X-Content-Type-Options`, `Referrer-Policy`, `Permissions-Policy`**
  cannot be set on Pages **by any means**.

Do not read `/_headers` as something Pages applies — **it does not**. Pages
publishes it as an ordinary static text file and ignores its contents.

> Note on HSTS: the `*.github.io` apex is already HSTS-preloaded by GitHub, so
> the site is HTTPS-only there regardless. The `_headers` HSTS line (with
> `preload`) is meant for a future **custom domain** (e.g. `achetiq.org.ar`).
> Only keep `preload` once every subdomain of that domain is HTTPS-only — it is
> hard to undo.

## Where/how to apply the full set later

**Cloudflare Pages or Netlify** — both consume `/_headers` natively. Deploy the
repo there and the full header set (CSP + `frame-ancestors` + HSTS + the rest) is
served as-is. No code change needed.

**Cloudflare CDN in front of GitHub Pages** — `_headers` is not read in this
mode; add a *Transform Rule → Modify Response Header* (or a small Worker) with
the same values:

```
# Cloudflare Transform Rule (Modify Response Header → Set static), one per header:
Content-Security-Policy:    default-src 'self'; base-uri 'self'; object-src 'none'; script-src 'self'; style-src 'self'; img-src 'self'; font-src 'self'; connect-src 'self'; frame-src https://www.google.com; form-action 'self'; frame-ancestors 'none'; upgrade-insecure-requests
Strict-Transport-Security:  max-age=63072000; includeSubDomains; preload
X-Frame-Options:            DENY
X-Content-Type-Options:     nosniff
Referrer-Policy:            strict-origin-when-cross-origin
Permissions-Policy:         camera=(), microphone=(), geolocation=(), browsing-topics=(), payment=(), usb=()
Cross-Origin-Opener-Policy: same-origin
```

When applied behind a real proxy, this set is expected to score **A / A+** on
[securityheaders.com](https://securityheaders.com) and a high grade on Mozilla
Observatory. From the restricted CI/agent network the deployed `*.github.io`
origin is **not reachable** (egress policy returns 403), so that grade must be
captured from an unrestricted network once a proxy is in place.

## Keeping CSP in sync

The CSP string appears in two places that **must match**: the `<meta>` (source of
truth: the `CSP` constant in `scripts/build-csp.mjs`) and `/_headers` (which adds
only `frame-ancestors 'none'`). If you change one, change the other in the same
commit. The two differ **only** by `frame-ancestors`, which a `<meta>` cannot
carry.
